# Помощник поддержки Haven

## Назначение

Эта задача добавляет проверяемого помощника поддержки для компании Haven.
Помощник отвечает на типовые вопросы клиентов о домашней охране, устройствах,
тарифах наблюдения, оплате, доступе к учетной записи, настройке устройств и
простом поиске неисправностей.

Главная цель реализации: не дать помощнику выходить за границы поддержки.
Поэтому сначала описаны правила, разрешенные темы, запрещенные темы и случаи,
которые нужно передавать человеку. После этого добавлены данные, серверная
логика, страница в приложении и проверки.

Важное решение: в этой версии не используется вызов внешней языковой модели.
Поведение сделано через явные правила и файл знаний. Это проще проверять:
одинаковый вход всегда дает одинаковый результат, а опасные темы не уходят в
свободную генерацию текста.

## Границы Поддержки

Помощник отвечает только в пределах поддержки Haven. Границы записаны в
`boundary_rules` внутри `data/haven_support_knowledge.json`.

Разрешенные темы:

- настройка и привязка устройств Haven;
- общие вопросы об оплате и счетах;
- общие вопросы о тарифах наблюдения;
- простые действия при неисправностях устройств.

Темы для передачи человеку:

- доступ к учетной записи;
- сброс пароля;
- двухфакторная проверка;
- смена почты;
- спорная оплата;
- возврат денег;
- отмена или понижение тарифа;
- активная тревога, пожар, вторжение или другой риск безопасности;
- неисправность устройства после уже выполненных простых шагов;
- замена устройства или гарантийный случай.

Темы для отказа:

- поддержка устройств других компаний;
- юридические, страховые и медицинские советы;
- запросы не по домашней охране Haven;
- любые действия, где помощник мог бы изменить учетную запись, оплату,
  тариф или обещать вызов служб.

## Файл Знаний

Основной файл:

```text
data/haven_support_knowledge.json
```

Он состоит из нескольких разделов.

### Термины Haven

Раздел `canonical_terms` задает единые названия продуктов:

- `Haven Cam` — камера Haven;
- `Haven Hub` — домашняя станция;
- `Haven Secure` — платный тариф наблюдения;
- `Haven Sense` — датчики двери, окна, движения, дыма и воды.

Эти термины нужны, чтобы ответы использовали одинаковые названия, а не мешали
разные варианты формулировок.

### Правила Границ

Раздел `boundary_rules` хранит общие запреты и разрешения:

- отвечать только по темам поддержки Haven;
- не проверять личность клиента;
- не менять учетную запись;
- не менять оплату;
- не отменять услуги;
- не выдавать возвраты;
- не обещать вызов экстренных служб;
- передавать опасные и учетные случаи человеку.

Эти правила возвращаются в ответе, чтобы вызывающая часть могла показать или
сохранить причину поведения помощника.

### Разрешенные Запросы

Раздел `supported_requests` описывает запросы, на которые можно отвечать сразу.
Для каждого запроса указаны:

- `intent` — внутреннее имя темы;
- `label` — человекочитаемое название;
- `route` — действие, здесь это `answer`;
- `keywords` — ключевые слова для распознавания;
- `entry_ids` — ссылки на записи знаний.

Поддерживаются четыре темы:

| Внутреннее имя | Что означает | Что делает помощник |
|---|---|---|
| `device_setup` | Настройка устройств | Дает шаги привязки и установки |
| `billing_general` | Общие вопросы оплаты | Объясняет, где смотреть счета и способ оплаты |
| `monitoring_plans` | Тарифы наблюдения | Объясняет разницу между базовым и платным тарифом |
| `basic_troubleshooting` | Простые неисправности | Дает базовые шаги проверки питания, связи и приложения |

### Запросы Для Передачи Человеку

Раздел `escalation_requests` описывает темы, где автоматический ответ
недостаточен или рискован.

Для каждой темы указаны:

- `intent` — внутреннее имя;
- `route` — действие, здесь это `escalate`;
- `keywords` — признаки темы в сообщении;
- `reason` — почему нужен человек;
- `next_step` — что должен сделать следующий сотрудник поддержки.

Темы передачи:

| Внутреннее имя | Когда срабатывает | Причина |
|---|---|---|
| `account_access` | вход, пароль, двухфакторная проверка, смена почты | нужна проверка личности |
| `safety_sensitive` | тревога, пожар, вторжение, полиция, срочная опасность | нельзя решать только автоматическим ответом |
| `billing_sensitive` | возврат, спор, отмена, ошибочное списание | нужно смотреть учетную запись и историю оплат |
| `unresolved_device_failure` | устройство все еще не работает, нужна замена или гарантия | нужна глубокая диагностика |

### Запросы Для Отказа

Раздел `unsupported_requests` описывает темы, на которые помощник не должен
отвечать по существу.

Примеры:

- устройства других компаний;
- юридические вопросы;
- страховые вопросы;
- медицинские вопросы;
- вопросы не по поддержке Haven.

Для таких случаев помощник возвращает действие `refuse` и короткое объяснение.

### Записи Знаний

Раздел `knowledge` содержит готовые короткие ответы и шаги.

Каждая запись содержит:

- `id` — ключ записи;
- `intent` — тема;
- `title` — название источника;
- `answer` — ответ клиенту;
- `steps` — рабочие шаги;
- `escalate_when` — условия, когда нужно передать человеку.

Записи сделаны короткими и отдельными. Это важно: маленькие записи легче
проверять, дополнять и заменять без риска сломать другие темы.

### Сценарии Разговора

Раздел `flows` задает порядок действий для каждой поддерживаемой темы.

Пример для настройки устройства:

1. определить тип устройства;
2. дать шаги привязки;
3. проверить подтверждение в приложении;
4. передать человеку, если привязка не проходит.

Эти шаги возвращаются в поле `next_steps`.

## Как Работает Серверная Часть

Серверная часть разделена на маленькие файлы.

### `src/llm/haven_support_models.py`

Здесь описаны внутренние структуры данных:

- `HavenSupportSource` — источник ответа;
- `HavenHandoffPayload` — данные для передачи человеку;
- `HavenSupportOutput` — итог работы помощника.

Эти структуры отделены от HTTP-слоя. Так основную логику можно проверять без
запуска сервера.

### `src/llm/haven_support_config.py`

Файл отвечает за загрузку `data/haven_support_knowledge.json`.

После загрузки выполняется проверка:

- нет повторяющихся `id` в записях знаний;
- каждая разрешенная тема ссылается на существующую запись знаний;
- для каждой разрешенной темы есть сценарий разговора.

Если файл знаний испорчен, ошибка появится сразу при создании помощника, а не
после случайного обращения клиента.

### `src/llm/haven_support_bot.py`

Это главный файл поведения.

Метод:

```text
HavenSupportBot.respond(message, channel)
```

Он делает следующие шаги:

1. Убирает лишние пробелы из сообщения.
2. Проверяет, что сообщение не пустое.
3. Проверяет канал: разрешены только `help_center` и `in_app`.
4. Приводит сообщение к простой форме: нижний регистр, пробелы вместо знаков,
   одинарные пробелы.
5. Ищет подходящее правило.
6. Возвращает ответ, передачу человеку, отказ или просьбу уточнить вопрос.

Порядок выбора правила важен:

1. сначала проверяются темы для передачи человеку;
2. затем темы для отказа;
3. затем темы, на которые можно ответить;
4. если совпадений нет, возвращается просьба уточнить вопрос.

Такой порядок защищает опасные случаи. Например, если сообщение содержит и
слова про устройство, и слова про вторжение, сработает передача человеку, а не
обычная инструкция по устройству.

### `src/llm/haven_support_api.py`

Этот файл связывает внутреннюю логику с HTTP-ответом.

Он описывает:

- входной запрос `HavenSupportRequest`;
- источник ответа `HavenSupportSourceResponse`;
- данные передачи человеку `HavenHandoffPayloadResponse`;
- полный ответ `HavenSupportResponse`.

Если входные данные неверные, сервер возвращает ошибку `400`.

### `src/llm/api.py`

Здесь путь подключен к общему приложению:

```text
POST /support/haven/respond
```

Путь принимает сообщение клиента и канал, вызывает `respond_to_haven_support`,
а затем возвращает структурированный ответ.

## Как Выбирается Ответ

Внутри выбора ответа используется простой подсчет совпадений ключевых слов.

Для каждой группы правил берутся ключевые слова из файла знаний. Если ключевое
слово найдено в нормализованном сообщении, правило получает балл. Правило с
самым большим баллом внутри текущей группы считается найденным.

Группы проверяются не одновременно, а по очереди:

```text
передача человеку -> отказ -> обычный ответ -> уточнение
```

Это сделано намеренно. Передача человеку важнее обычного ответа, потому что
учетные, платежные и опасные случаи нельзя закрывать только автоматическим
сообщением.

## Виды Ответов

Сервер возвращает одно из четырех действий.

### `answer`

Обычный ответ по разрешенной теме.

Пример: клиент спрашивает, как привязать камеру Haven. Помощник возвращает:

- тему `device_setup`;
- действие `answer`;
- текст с шагами;
- список следующих шагов;
- источник `device_setup`;
- без данных передачи человеку.

### `escalate`

Передача человеку.

Пример: клиент пишет, что не может войти и хочет сбросить пароль. Помощник
возвращает:

- тему `account_access`;
- действие `escalate`;
- объяснение, что нужен сотрудник поддержки;
- причину передачи;
- следующий шаг для сотрудника;
- исходный текст клиента как контекст.

### `refuse`

Отказ отвечать по существу.

Пример: клиент просит настроить камеру другой компании. Помощник отвечает, что
помогает только с продуктами Haven.

### `clarify`

Просьба уточнить вопрос.

Это используется, когда сообщение не совпало ни с одной темой, но и не попало в
явно запрещенную область. Помощник перечисляет, по каким направлениям он может
помочь.

## Форма Запроса И Ответа

Запрос:

```json
{
  "message": "How do I pair a Haven Cam?",
  "channel": "help_center"
}
```

Поле `channel` принимает:

- `help_center` — центр помощи;
- `in_app` — встроенная помощь в приложении.

Ответ:

```json
{
  "message": "How do I pair a Haven Cam?",
  "normalized_message": "how do i pair a haven cam",
  "channel": "help_center",
  "intent": "device_setup",
  "action": "answer",
  "answer": "Open the Haven app, choose Add device...",
  "next_steps": ["Identify device type", "Give pairing steps"],
  "boundary_rules": ["Answer only Haven home security product..."],
  "sources": [
    {
      "source_id": "device_setup",
      "title": "Set up Haven devices",
      "intent": "device_setup"
    }
  ],
  "handoff": null
}
```

При передаче человеку поле `handoff` заполняется:

```json
{
  "intent": "account_access",
  "status": "needs_human",
  "context": "I can't log in and need a password reset",
  "reason": "Requires identity verification or account change.",
  "next_step": "Human support verifies identity before changing account access."
}
```

## Почему Нет Вызова Языковой Модели

В этой задаче главное — границы поддержки. Для таких задач свободная генерация
опасна без дополнительных проверок: помощник может пообещать действие, дать
совет вне правил или закрыть случай, который должен смотреть человек.

Поэтому первая версия сделана на явных правилах:

- легче проверить;
- легче объяснить результат;
- нет сетевого вызова;
- нет зависимости от ключей доступа;
- одинаковое сообщение дает одинаковое действие;
- опасные темы можно поставить выше обычных ответов.

Языковую модель можно добавить позже, но только после того, как уже есть
проверенная политика, набор контрольных вопросов и строгий слой проверки
ответов.

## Как Дополнять Помощника

Чтобы добавить новую разрешенную тему:

1. Добавить запись в `knowledge`.
2. Добавить правило в `supported_requests`.
3. Добавить сценарий в `flows`.
4. Добавить проверку в `tests/test_haven_support_bot.py`.
5. Запустить точечные проверки.

Чтобы добавить новую тему для передачи человеку:

1. Добавить правило в `escalation_requests`.
2. Указать `reason` и `next_step`.
3. Добавить проверку, что тема возвращает `action = "escalate"`.

Чтобы добавить новую тему для отказа:

1. Добавить правило в `unsupported_requests`.
2. Указать безопасный текст отказа.
3. Добавить проверку, что тема возвращает `action = "refuse"`.

## Текущие Ограничения

- Данные Haven учебные, а не реальные рабочие правила компании.
- Тексты ответов пока на английском языке, потому что так создан демонстрационный
  набор знаний.
- Поиск темы идет по ключевым словам, без понимания синонимов и сложных фраз.
- Помощник не входит в учетную запись клиента.
- Помощник не меняет оплату, тарифы, почту или пароль.
- Помощник не вызывает экстренные службы.
- Нет связи с настоящей системой поддержки и очередью сотрудников.
- Нет журнала обращений клиентов.
- Нет измерения качества на большом наборе реальных вопросов.

## Что Нужно Для Рабочего Использования

Перед настоящим использованием нужно:

- заменить учебные правила реальными правилами Haven;
- утвердить тексты ответов с командой поддержки и юристами;
- добавить больше примеров клиентских вопросов;
- добавить проверку качества по темам, отказам и передачам человеку;
- подключить настоящую систему поддержки для передачи обращения;
